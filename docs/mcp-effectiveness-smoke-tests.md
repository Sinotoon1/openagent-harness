# MCP Effectiveness Smoke Tests

This document measures whether the MCP harness helps compared with sending the
same model/provider/prompt directly without harness repair, routing, and
telemetry tools. It is an internal smoke-test guide, not a production eval
pipeline.

## Comparison Method

Run each task twice:

- Baseline without MCP: call the same provider/model directly or use a client path
  that bypasses this harness.
- With MCP harness: call the equivalent MCP tool through
  `oss-agent-harness-mcp`.

Hold these inputs constant wherever possible:

- same canonical model ID and provider-native slug
- same provider account and base URL
- same prompt and tool schema
- same `sessionId` for harness runs
- same temperature and max-token settings
- same provider capability request

Record the result, retry count, total elapsed time, time to first token when the
client can observe it, final answer quality notes, and sanitized telemetry.

## Measures

| Measure | What to Record | Useful Direction |
|---|---|---|
| Repair success rate | Repaired calls divided by invalid calls submitted to `repair_tool_input` | Higher |
| Invalid tool call reduction | Invalid calls after repair compared with baseline invalid calls | Lower |
| Retry reduction | Provider/tool retries needed to complete the same task | Lower |
| TTFT improvement | Time to first token for equivalent real-provider chat paths | Lower |
| Provider fallback correctness | Fallback only before meaningful assistant output; no fallback after output starts | Correct behavior |
| JSONL telemetry correctness | One sanitized event per JSONL line, parseable JSON, expected event types | Correct behavior |
| Secret leakage | Raw API keys, headers, env values, prompts, file contents, and provider raw payloads in outputs/telemetry | None |

## Token Overhead

Track token overhead separately from provider-reported usage:

- MCP tool schema tokens: tool descriptions and JSON schema exposed to the model.
- MCP tool call argument tokens: JSON arguments the model emits for a tool call.
- MCP tool response tokens: content returned from MCP tools to the model/client.
- Provider prompt tokens: tokens sent to the provider model.
- Provider completion tokens: tokens generated by the provider model.

Approximate token count when exact tokenizer data is unavailable:

```text
estimatedTokens = Math.ceil(charCount / 4)
```

This is only an estimate. Compare overhead against avoided retries, avoided
provider HTTP failures, and avoided repeated tool-call attempts. A small MCP
response can still be worthwhile if it prevents a full model retry.

The optional local helper can estimate JSON payload size without external
dependencies:

```powershell
node scripts/estimate-json-tokens.mjs path\to\payload.json
Get-Content path\to\payload.json -Raw | node scripts/estimate-json-tokens.mjs
```

## Smoke Tests

| Smoke Test | Tool | Expected Result | Effectiveness Signal |
|---|---|---|---|
| Bare string to array | `repair_tool_input` with `schemaName: "pathBatch"` and `input: "src/server.ts"` | Repaired to an array shape; validation passes | Avoids one invalid tool retry |
| JSON array string to array | `repair_tool_input` with `input: "[\"src/server.ts\"]"` | Parses JSON array string into array | Avoids manual retry or prompt correction |
| Markdown path unwrap | `repair_tool_input` using a path field containing a markdown link | Unwraps to plain path string | Reduces path-format tool failures |
| Dangerous descriptor rejection | `repair_tool_input` with a descriptor containing `__proto__`, `prototype`, or `constructor` path/key | Structured invalid response; no repair applied | Confirms descriptor boundary |
| Policy inspection | `inspect_model_policies` with `modelId: "deepseek-v4-pro"` | Sanitized policy summary and warnings | Confirms model policy visibility without YAML edits |
| Policy doctor | `run_policy_doctor` with provider config included | Read-only health report; no live credential checks | Finds config/policy issues without account probing |
| Telemetry query | `query_telemetry` with a bounded `limit` | Sanitized telemetry; no raw metadata secrets | Confirms telemetry response boundary |
| Harness stats | `get_harness_stats` | Aggregate local health stats | Confirms quick trend visibility |
| Insufficient data suggestions | `suggest_repair_policy` with a model that has no repaired events | `policySuggestions` row with `status: "insufficient_data"` | Avoids overclaiming from empty telemetry |
| Real provider chat | `oss_chat` with `deepseek-v4-pro`, stable `sessionId`, and configured `deepseekPrimary` | Normal and streaming calls pass; raw preview is sanitized when requested | Confirms provider path is useful under real conditions |

## JSONL Telemetry Check

For a local JSONL smoke run:

```powershell
$env:OSS_HARNESS_TELEMETRY_SINK = "jsonl"
$env:OSS_HARNESS_TELEMETRY_JSONL_PATH = "D:\logs\oss-agent-harness\telemetry.jsonl"
```

After running smoke tests, verify:

- every line parses as JSON
- event type names match expected harness events
- `sessionIdHash` is present instead of raw `sessionId`
- metadata is bounded and sanitized
- no provider API key, authorization header, env dump, prompt body, or raw file
  content appears
